# CEF Example

This example generates Haskell bindings for the
[Chromium Embedded Framework (CEF)](https://bitbucket.org/chromiumembedded/cef)
C API using `hs-bindgen`.

## Purpose

CEF is a complex, real-world C API with features that stress-test `hs-bindgen`:

* Huge type definitions (`cef_types.h` is ~117KB)
* Structs with embedded function pointers (virtual method tables)
* Reference counting via `cef_base_ref_counted_t`
* Platform-specific type aliases
* Hundreds of enums and nested structs
* 79 C API headers with deep dependency chains (11 levels)

This example is exploratory: its goal is to surface any unforeseen difficulties
with CEF's header structure.

## Prerequisites

* GHC 9.6.x (or compatible)
* Cabal 3.16.x (or compatible)
* LLVM/Clang 15 (or compatible)
* `curl` (for downloading the CEF binary distribution)
* `tar` with bzip2 support

No need to build Chromium from source. The script downloads a pre-built binary
distribution (~287MB) from the Spotify CDN, which includes both headers and
`libcef.so`.

### NixOS

On NixOS, `libcef.so` depends on many system libraries (glib, X11, nss, etc.)
that are not in the default library path. Use a `nix-shell` expression that sets
up `LD_LIBRARY_PATH`:

```bash
nix-shell -E '
  with import <nixpkgs> {};
  let libs = [
    glib nss nspr cups.lib xorg.libX11 xorg.libXcomposite
    xorg.libXdamage xorg.libXext xorg.libXfixes xorg.libXrandr
    libgbm expat libxkbcommon cairo pango systemd alsa-lib
    at-spi2-core dbus xorg.libxcb
  ]; in mkShell {
    buildInputs = libs;
    shellHook = "export LD_LIBRARY_PATH=${lib.makeLibraryPath libs}:$LD_LIBRARY_PATH";
  }
'
```

Then run `./generate-and-run.sh` from within that shell.

## Running

```bash
cd examples/cef
./generate-and-run.sh
```

The script will:

1. Download the CEF binary distribution (if not already present)
2. Generate Haskell bindings for all 79 C API headers (in dependency order)
3. Update `cabal.project.local` with include/library paths
4. Build and run a minimal Haskell program that loads the bindings

The dependency order was determined using:

```bash
cabal run hs-bindgen-cli -- info include-graph -I "$CEF_ROOT" "include/capi/cef_app_capi.h"
```

combined with a topological sort of the resulting graph.

## CEF version

The example pins CEF version `145.0.28+g51162e8+chromium-145.0.7632.160`
(linux64 minimal distribution).

## Results

* **78 out of 79** C API headers generate bindings successfully
* **193 Haskell modules** are produced (main + FunPtr/Safe/Unsafe variants)
* All generated code compiles and links against `libcef.so`
* The test executable initializes CEF, creates objects via the C API, and
  exercises vtable dispatch (function pointer calls through generated structs)

Expected output (stderr warnings from CEF are normal):

```
CEF initialized successfully.
  API version: 14500
  cef_command_line_create: OK
  is_valid: 1
  has_switches (empty): 0
All CEF binding calls succeeded.
CEF shutdown complete.
```

## Issues log

This section documents issues encountered while developing the bindings.

### 1. `cef_process_util_capi.h`: missing `#include` (CEF translator bug)

`cef_process_util_capi.h` is auto-generated by the CEF translator tool. It
declares `cef_launch_process(struct _cef_command_line_t* command_line)` but only
includes `cef_base_capi.h`, missing the required `#include` for
`cef_command_line_capi.h`. Because the struct tag first appears inside a function
prototype, it has function prototype scope (C11 6.2.1p4) — making it a different
type from the file-scope `struct _cef_command_line_t` defined in
`cef_command_line_capi.h`. GCC/Clang compile this with a warning, but
`hs-bindgen` panics:

```
PANIC!: the impossible happened
A haskell identifier is not available: The C name of the declaration was not
mangled because it only appears in an underlying type.
```

The script skips this header automatically. This is a bug in CEF's C API
translator — the other 78 auto-generated headers all include the correct
dependencies.

### 2. API version must be configured via `cef_api_hash`

CEF 145 introduced an API versioning system. `libcef.so` is compiled with
`BUILDING_CEF_SHARED`, which includes ALL struct fields from ALL API versions.
Client code must call `cef_api_hash(CEF_API_VERSION, 0)` **before any other CEF
function** to tell `libcef.so` which API version the client was compiled with.
Without this call, CppToC struct wrappers crash with SIGTRAP when creating
objects (e.g. `cef_command_line_create`).

The bindings are generated with `-D CEF_API_VERSION=14500` and the test
executable calls `cef_api_hash(14500, 0)` before `cef_execute_process`.

Note: `cef_api_hash` is declared in `include/cef_api_hash.h` (not under
`include/capi/`), so it is not auto-generated by the CEF translator. The
generation script handles these non-capi headers separately.

### 3. CEF runtime dependencies

CEF is a full browser engine and `libcef.so` depends on many system libraries
(glib, nspr, nss, X11, dbus, cups, etc.). On Ubuntu these are standard system
packages. On NixOS, see the nix-shell instructions above.
