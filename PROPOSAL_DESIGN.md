# High-level binding DSL for `hs-bindgen` — proposal

## Abstract

We propose a small, type-safe, compositional DSL for lifting the
low-level FFI bindings generated by `hs-bindgen` into ergonomic
high-level Haskell wrappers. Each position in a C signature is
described by a value-level marshaller of one of three roles (input,
output, return); a binding spec is a three-constructor GADT that
stitches those marshallers together; one interpreter,
`toHighLevel`, materialises the wrapper. The DSL is plain Haskell —
no preprocessor, no custom syntax — and lets the user pick
marshallers per call. The cost is a small amount of type-class
machinery (`Spread`, `ThreadIn`, `ThreadOut`) that the user never
writes but does occasionally see in error messages; that trade-off
is discussed honestly later in the doc.

## Goals and non-goals

The DSL aims to:

- **Customise per call**: a `String` can use `withCString` at one
  call site and `useAsCStringLen` at another, without inventing
  domain newtypes.
- **Make output parameters declarative**: no manual
  `alloca`/`peek` at user level; `output peekIntOut` allocates and
  reads back for you.
- **Compose without ceremony**: combinators chain with `$`, no new
  operator to learn.
- **Stay type-safe**: misuse is caught at compile time and the
  error messages point at the spec, not at internals.
- **Stay idiomatic**: no language extensions beyond what GHC2021
  ships plus `GADTs`, `FunctionalDependencies`, and
  `UndecidableInstances`.

The DSL does *not* aim to:

- Replace the round-1 `hl` Tier-3 inference combinator. The two are
  complementary; the DSL is the fully-explicit alternative.
- Auto-generate wrappers from C declarations. That is Milestone-3
  generator work; this DSL is the natural target.
- Produce a flat *N*-tuple result for *N* ≥ 3 outputs. We use
  right-associative nested pairs and document the awkwardness;
  see §[Open questions, risks, caveats](#open-questions-risks-caveats).
- Add `TypeError`-clad instances by default. Real GHC output is
  inspected in §[Reading error messages](#reading-error-messages);
  improvements are tracked as future work.

## Introduction and context

Currently, `hs-bindgen` reads a C header and generates low-level Haskell
bindings. In these bindings, a C `int` becomes `CInt`, a `const char *`
becomes `PtrConst CChar`, a flexible array becomes `IncompleteArray a`, etc.

However, what most users want is a Haskell-native API where strings are
`String` or `ByteString`, bools are `Bool`, nullable pointers are `Maybe`,
arrays are `Vector`. The QR-code example in `examples/c-qrcode/` shows what
the gap looks like in practice:

```haskell
basicDemo :: IO ()
basicDemo =
  F.withCAString "Hello, world!" $ \text ->
    F.allocaArray (fromIntegral QR.qrcodegen_BUFFER_LEN_MAX) $ \tempBuffer ->
      F.allocaArray (fromIntegral QR.qrcodegen_BUFFER_LEN_MAX) $ \qrCode -> do
        b <- QR.qrcodegen_encodeText
              (PtrConst.unsafeFromPtr text) tempBuffer qrCode
              QR.Qrcodegen_Ecc_LOW QR.qrcodegen_VERSION_MIN
              QR.qrcodegen_VERSION_MAX QR.Qrcodegen_Mask_AUTO
              (F.fromBool True)
        ...
```
(`examples/c-qrcode/hs-project/app/Main.hs:65-75`)

The shape of that block is dominated by plumbing: a `withCAString`
bracket around a `String`, two `allocaArray` brackets around the
buffers, a `PtrConst.unsafeFromPtr` to convert one pointer's
nullability story, a `F.fromBool` to convert a `Bool` to `CBool`. The
single line of real intent is the call to `qrcodegen_encodeText`.

What we would like instead is a wrapper whose Haskell signature reads
like a Haskell function:

```haskell
qrEncodeText :: String -> Bool -> IO (Maybe QRCode)
```

And whose body is short enough not to obfuscate the call. The plumbing
above is mechanical, so it would be nice if we could have some way of
abstracting it away.

This document proposes a small DSL that lets the user write the
plumbing declaratively, per call, in a few lines.

## State of the art: `c2hs`

The closest prior art in the Haskell ecosystem is [`c2hs`][c2hs-guide]:
a preprocessor that reads C headers and rewrites `{#fun ...#}` /
`{#enum ...#}` / `{#pointer ...#}` directives in a `.chs` file into a
Haskell module with `foreign import` declarations and matching
marshalling wrappers.

Two small examples to fix the idea:

```c
int do_thing(const char *str, int len);
int get_size(size_t *out);
```

In c2hs:

```haskell
{#fun do_thing as ^
   { withCStringLen* `String'& } -> `Int' #}

{#fun get_size as ^
   { alloca- `Int' peekIntConv* } -> `Int' #}
```

The first declaration says: take one Haskell `String` (the back-ticked
`String'`), use `withCStringLen` as its input marshaller (the `*` after
a marshaller name means "an `IO` bracket"), and splat the resulting
`(Ptr CChar, Int)` across two C arguments (the trailing `&` on the
type). The generated Haskell is:

```haskell
doThing :: String -> IO Int
doThing s = withCStringLen s $ \(p, n) ->
              fromIntegral <$> do_thing'_ p (fromIntegral n)
```

The second says: this argument is suppressed from the Haskell signature
(`alloca-`, the trailing `-` on the marshaller, means "don't expose
it"), the marshaller allocates a `size_t*`, and after the call we
`peekIntConv*` it into an `Int` (the trailing `*` again being IO,
appended to the result tuple). Generated:

```haskell
getSize :: IO Int
getSize = alloca $ \p -> get_size'_ p >> fromIntegral <$> peek p
```

The full grammar adds about six modifiers:

| modifier | meaning |
|---|---|
| `*`           | IO bracket (the marshaller runs in `IO`)                  |
| `-`           | suppress from the Haskell signature (used on input + output) |
| `&`           | one Haskell value to two C arguments                       |
| `*-`          | effect-only, e.g. error-check discard                      |
| `+` / `+S` / `+N` | allocate scratch and append it to the result tuple    |
| `%`           | struct passed by value, via a generated C wrapper          |

c2hs picks a sensible default marshaller per `(Hs type, C type)` pair,
so users typically write just `` `String' `` and get `withCString*` /
`peekCString*` for free.

**What c2hs gets right.** The decomposition of an argument position
into *(in marshaller, out marshaller, modifiers)* is the right
conceptual unit. Every binding boils down to this shape. The `&`
modifier serves a real, common pattern (`1 Hs → N C slots`) and
solves it for `N = 2`.

**What c2hs is less good at:**

- It is a preprocessor with its own non-Haskell syntax that GHC does
  not see;
- The `&` modifier is hard-coded to `N = 2`; arbitrary `N` is not expressible.
- Not type safe.

We want the same expressive power inside the language: combinators
the user can pass around, swap, abstract over, and reason about
without leaving Haskell. GHC should see every step of the
construction, so type errors point at the user's spec.

[c2hs-guide]: https://github.com/haskell/c2hs/wiki/User-Guide

## What we are aiming for

Take the C function:

```c
int parse_int(const char *str, int *out);
```

`hs-bindgen` emits the low-level binding:

```haskell
foreign import ccall unsafe "parse_int"
  parse_int :: PtrConst CChar -> Ptr CInt -> IO CInt
```

The wrapper we want is:

```haskell
hsParseInt :: String -> IO (Int, Int)
```

The argument is the input string; the result is the parsed integer
paired with the function's status code. Using the designed API, the
wrapper can be defined in three lines:

```haskell
hsParseInt :: String -> IO (Int, Int)
hsParseInt = toHighLevel ( input  withCStringIn                   -- 1st C arg: const char* ← String
                         $ output (peekOut (pure . fromIntegral)) -- 2nd C arg: int*        → Int
                         $ result (pureRes  fromIntegral)         -- C return:   int        → Int
                         ) parse_int
```

Three combinators chained with `$`, each describing one position in
the C signature in order. `toHighLevel` interprets the resulting spec
against the low-level callable and returns the high-level wrapper.

## Marshallers

Before introducing the library itself, let's see what values it consumes.
A *marshaller* is a value that carries a per-position conversion.
There are three roles a position in a C signature can play: input,
output, or return. And we give each one a small record type:

```haskell
data InMarshaller hs slot = InMarshaller
  { withIn :: forall r. hs -> (slot -> IO r) -> IO r }

data OutMarshaller hs slot = OutMarshaller
  { withOutM :: forall r. (slot -> IO r) -> IO (hs, r) }

newtype ResMarshaller c hs = ResMarshaller
  { runResM :: c -> IO hs }
```

Read the type variables as:

- `hs`: the Haskell-side type the user sees at the call site;
- `c`: the C-side type the foreign import returns;
- `slot`: the C-side shape the marshaller fills. For a single C
  argument that is one type; for the c2hs `&` case (one Haskell
  value spanning multiple C args) it's a `Slot<N>`
  constructor that names the arity.

Each record is shaped to fit its job:

- `InMarshaller` is a bracket: take the Haskell value, hand
  the continuation the C slot, tear down when the continuation
  returns.

  ```haskell
  withCStringIn :: InMarshaller String (PtrConst CChar)
  withCStringIn = InMarshaller $ \s k ->
    withCString s (k . PtrConst.unsafeFromPtr)
  ```

- `OutMarshaller` is the dual: it allocates a slot itself, runs
  the continuation, then *peeks* a value back. The peeked value is
  paired with the continuation's result:

  ```haskell
  allocaOut :: Storable c => OutMarshaller c (Ptr c)
  allocaOut = OutMarshaller $ \k -> alloca $ \p -> do
    r <- k p
    c <- peek p
    pure (c, r)

  peekOut :: Storable c => (c -> IO hs) -> OutMarshaller hs (Ptr c)
  peekOut f = OutMarshaller $ \k -> alloca $ \p -> do
    r  <- k p
    c  <- peek p
    hs <- f c
    pure (hs, r)
  ```

- **`ResMarshaller`** is the simplest: an effectful `c -> IO hs`.
  Two helpers cover the common cases:

  ```haskell
  pureRes :: (c -> hs) -> ResMarshaller c hs
  pureRes f = ResMarshaller (pure . f)

  ioRes :: (c -> IO hs) -> ResMarshaller c hs
  ioRes = ResMarshaller
  ```

### Marshallers that span multiple C arguments

For the c2hs `&` case, where one Haskell value corresponds to multiple
consecutive C arguments, we name the arity explicitly:

```haskell
data Slot2 a b     = Slot2 a b
data Slot3 a b c   = Slot3 a b c
data Slot4 a b c d = Slot4 a b c d
```

A `ByteString` marshaller using `useAsCStringLen` produces a
`Slot2 (PtrConst CChar) CSize`:

```haskell
useAsByteStringLenIn
  :: InMarshaller ByteString (Slot2 (PtrConst CChar) CSize)
useAsByteStringLenIn = InMarshaller $ \bs k ->
  BS.useAsCStringLen bs $ \(p, n) ->
    k (Slot2 (PtrConst.unsafeFromPtr p) (fromIntegral n))
```

The user never writes `Slot2` at the call site since it lives inside the
marshaller body. What matters at the call site is that `useAsByteStringLenIn`
consumes one Haskell value and two C arguments. The plumbing that turns `Slot2
p n` into `f p n` is the job of another type class: `Spread`.

### Nullable inputs

C pointers can be `NULL`. The Haskell-side analogue is `Maybe a`:
`Nothing` is the slot's null value, `Just x` delegates to the
wrapped input marshaller. A tiny helper class names the "this slot
has a null" capability:

```haskell
class Nullable slot where
  nullValue :: slot

instance Nullable (Ptr a)      where nullValue = nullPtr
instance Nullable (PtrConst a) where nullValue = PtrConst.unsafeFromPtr nullPtr
```

Wrapping an existing input marshaller in `nullableIn` makes it
accept `Maybe`:

```haskell
nullableIn :: Nullable slot => InMarshaller hs slot -> InMarshaller (Maybe hs) slot
nullableIn (InMarshaller m) = InMarshaller $ \mhs k -> case mhs of
  Nothing -> k nullValue
  Just hs -> m hs k
```

The `Nullable slot` constraint rules out non-pointer slots: a
`nullableIn pureIntIn` is a compile error because `CInt` has no
designated null, which is the right behaviour — there is no sensible
way to encode `Nothing` as a `CInt`. The constraint also rules out
multi-arg slots (`Slot2`, `Slot3`, ...) for the same reason.

A call-site example. Suppose:

```c
bool check_present(const char *name);
/* name may be NULL; returns false in that case. */
```

The wrapper accepts `Maybe String`:

```haskell
hsCheckPresent :: Maybe String -> IO Bool
hsCheckPresent = toHighLevel ( input  (nullableIn withCStringIn)
                             $ result (pureRes toBool)
                             ) check_present
  where toBool = (/= 0)
```

`Nothing` maps to the C-side null pointer; `Just s` runs
`withCString` as usual. The same `nullableIn` works for any
single-pointer-slot marshaller.

## The high level design

In order for the designed library to be able to represent the c2hs
capabilities, it needs to solve the following constraints and capture
the following information:

- Make it possible to specify marshalling functions, for:
  - input parameters
  - output parameters (which become results on the Haskell side)
  - results
- Map one Haskell value to multiple C slots (the c2hs `&` case):
  `ByteString` maps to a `(PtrConst CChar, CSize)` C slot, a `Vector` maps
  to a `(Ptr a, CSize)` C slot, etc.
- Some type level mechanism that walks a user's high-level signature and the
  low-level binding's signature together, picking the right conversion
  per slot.

With this in mind the following DSL was created (ignore the type class
constraints for now):

```haskell
data Spec lo hi where
  Done :: ResMarshaller c hs
       -> Spec (IO c) (IO hs)

  In   :: (Spread slot lo lo', ThreadIn slot rest)
       => InMarshaller hs slot
       -> Spec lo' rest
       -> Spec lo (hs -> rest)

  Out  :: (Spread slot lo lo', ThreadOut slot hs rest rest')
       => OutMarshaller hs slot
       -> Spec lo' rest
       -> Spec lo  rest'
```

This Haskell GADT captures a lot of information, but let's start with its type
variables:

- `lo` corresponds to the low-level binding's type signature;
- `hi` corresponds to the high-level type signature, i.e. the goal.

Regarding the constructors:

- `Done` closes the spec. Note that the `lo` and `hi` type variables are
  instantiated to `IO c` and `IO hs` respectively.
- `In` wraps an inner spec by adding one input position argument
  at the head of the C signature (This is `Spread`s type class constraint's
  doing). The user-side type grows by one argument: if the inner spec produces
  `rest`, the outer spec produces `hs -> rest`.
- `Out` wraps an inner spec by adding one output position at the head of the
  C signature (This is `ThreadOut`'s type class constraint's doing). The
  user-side type does not gain an argument; instead the output's value joins
  the result tuple at the innermost `IO` (via `ThreadOut` constraint).

A `Spec lo hi` is the recipe. It knows what to do but hasn't met the low-level
function yet. The interpreter that brings the two together is
`toHighLevel`:

```haskell
toHighLevel :: Spec lo hi -> lo -> hi
toHighLevel (Done (ResMarshaller k))     cFn = cFn >>= k
toHighLevel (In  (InMarshaller  m) spec) cFn = \hs -> threadIn  (m hs) $ \slot ->
                                                  toHighLevel spec (spread slot cFn)
toHighLevel (Out (OutMarshaller m) spec) cFn =        threadOut  m     $ \slot ->
                                                  toHighLevel spec (spread slot cFn)
```

Three pattern-match branches, one per constructor. Pattern-matching on the
GADT brings the `Spread`/`ThreadIn`/`ThreadOut` constraints into scope inside
each branch, which is what lets `spread`, `threadIn`, and `threadOut`
typecheck. The interesting work is hidden behind those three names.

It is a bit hard to visualise all the heavy lifting that the type class
constraints are responsible, so let's introduce them:

### `Spread`

`Spread` is the bridge between a slot value and the C arguments it
represents. For a single-type slot, "spreading" is just function
application; for a `Slot<N>` slot, it's *N* applications in
a row.

```haskell
class Spread slot lo lo' | slot lo -> lo' where
  spread :: slot -> lo -> lo'

instance {-# OVERLAPPABLE #-} Spread t (t -> lo) lo where
  spread t f = f t

instance Spread (Slot2 a b)     (a -> b -> lo)           lo where
  spread (Slot2 a b)     f = f a b
instance Spread (Slot3 a b c)   (a -> b -> c -> lo)      lo where
  spread (Slot3 a b c)   f = f a b c
instance Spread (Slot4 a b c d) (a -> b -> c -> d -> lo) lo where
  spread (Slot4 a b c d) f = f a b c d
```

Read the class arguments as: given a `slot`, apply it to a (low-level)
function `lo` and return whatever `lo` becomes after consuming the slot's C
arguments.

Concrete example. Suppose:

```haskell
c_strncmp :: PtrConst CChar -> PtrConst CChar -> CSize -> IO CInt
```

and we have a `Slot2 pStr len :: Slot2 (PtrConst CChar) CSize`
from `useAsByteStringLenIn`. The `Slot2` instance fires:

```haskell
spread (Slot2 pStr len) c_strncmp
  = c_strncmp pStr len
  :: PtrConst CChar -> IO CInt
```

The slot consumed two C arguments. What's left is a function expecting one
more `PtrConst CChar`.

The single-arg fallback fires for plain slots. Spreading a `PtrConst
CChar` from `withCStringIn` over the same function:

```haskell
spread (PtrConst.unsafeFromPtr p :: PtrConst CChar) c_strncmp
  = c_strncmp (PtrConst.unsafeFromPtr p)
  :: PtrConst CChar -> CSize -> IO CInt
```

One C argument consumed; two remain.

### `ThreadIn`

`ThreadIn` is what makes the right-to-left composition story work for input
arguments.

A marshaller bracket has CPS shape: it expects a continuation that ends in `IO
r`. But by the time `In` is wrapping an inner spec, the inner spec's
high-level type is already something of the following shape:

```
arg1 -> arg2 -> ... -> argN -> IO X
```

with *N* leading function arrows. The bracket needs to fire at the
innermost `IO X`, not at the outer `arg1 ->`. We have to introduce
fresh lambdas for `arg1, ..., argN` before the bracket runs, then
call the bracket's continuation with all those arguments in scope.

That walk is structural recursion on `(->)`. Each step peels off one
arrow; the base case fires the bracket at `IO X`:

```haskell
class ThreadIn slot hi where
  threadIn :: (forall r. (slot -> IO r) -> IO r)
           -> (slot -> hi) -> hi

-- Base case: hi is `IO r`. Fire the bracket now.
instance ThreadIn slot (IO r) where
  threadIn br f = br f

-- Step case: hi is a function. Introduce a fresh lambda for arg;
-- recurse on the remaining shape with arg already supplied.
instance ThreadIn slot rest
       => ThreadIn slot (arg -> rest) where
  threadIn br f = \arg -> threadIn br (\s -> f s arg)
```

Read the type arguments as:

- `slot`: the C-side value the bracket will hand to its
  continuation.
- `hi`: the inner spec's high-level shape. Always ends in some `IO X`; any
  leading `arg ->` arrows come from prior `input`s.

#### Worked example

Let's see this in motion. Take `hsStrncmp`:

```c
int strncmp(const char *s1, const char *s2, size_t n);
```

```haskell
hsStrncmp :: String -> ByteString -> IO Int
hsStrncmp = toHighLevel ( input  withCStringIn
                        $ input  useAsByteStringLenIn
                        $ result (pureRes fromIntegral)
                        ) c_strncmp
```

Build the spec innermost-first:

```
1) result (pureRes fromIntegral)
     :: Spec (IO CInt) (IO Int)

2) input useAsByteStringLenIn $ <1>
     The ByteString marshaller consumes two C arguments via its
     Slot2 slot. ThreadIn fires its base case because the
     inner spec's hi is `IO Int`.
     :: Spec (PtrConst CChar -> CSize -> IO CInt) (ByteString -> IO Int)

3) input withCStringIn $ <2>
     The String marshaller consumes one C argument. ThreadIn now
     sees an inner hi of `ByteString -> IO Int`, a function type.
     Its step instance fires, introducing a `\bs ->` lambda.
     :: Spec (PtrConst CChar -> PtrConst CChar -> CSize -> IO CInt)
             (String -> ByteString -> IO Int)
```

At runtime, `toHighLevel` unfolds the spec:

```haskell
toHighLevel <step3> c_strncmp
  = \s -> threadIn (withIn withCStringIn s)
                   (\slot -> toHighLevel <step2> (spread slot c_strncmp))
```

The body has type `PtrConst CChar -> (ByteString -> IO Int)`. The
step instance of `threadIn` fires once (one `\bs ->` introduced),
then the base case fires when the recursion reaches `IO Int`:

```haskell
  = \s -> \bs -> withIn withCStringIn s $ \slot ->
                   toHighLevel <step2> (spread slot c_strncmp) bs
  = \s -> \bs -> withCString s $ \pStr ->
                   toHighLevel <step2> (c_strncmp (PtrConst.unsafeFromPtr pStr)) bs
```

The `withCString` bracket fires *after* the user supplies both `s`
and `bs`.

### `ThreadOut`

`ThreadOut` is the dual of `ThreadIn`. Same recursion shape, but the
bracket's continuation also produces a Haskell value that has to be
threaded into the result tuple at the right depth.

```haskell
class ThreadOut slot hs hi hi' | slot hs hi -> hi' where
  threadOut :: (forall r. (slot -> IO r) -> IO (hs, r))
            -> (slot -> hi) -> hi'

-- Base case: hi is `IO r`. Fire the bracket; the result is `IO (hs, r)`.
instance ThreadOut slot hs (IO r) (IO (hs, r)) where
  threadOut br f = br f

-- Step case: hi is a function. Walk past arg, recurse, then introduce
-- a fresh lambda on the way back through.
instance ThreadOut slot hs rest rest'
       => ThreadOut slot hs (arg -> rest) (arg -> rest') where
  threadOut br f = \arg -> threadOut br (\s -> f s arg)
```

The two new type arguments are:

- `hs`: the Haskell value the bracket produces alongside the
  continuation's result. For an `Int` out-parameter via `peekOut`,
  `hs = Int`.
- `hi'`: what `hi` becomes once `hs` is prepended at its innermost
  `IO`.

#### Worked example

Here is the case the casual reader most often gets wrong: an output
position that appears *before* an input in the C signature.

```c
int compute(int *out, int n);   /* writes n*2 to *out, returns 0 */
```

```haskell
hsCompute :: Int -> IO (Int, Int)
hsCompute = toHighLevel ( output (peekOut (\(CInt v) -> pure (fromIntegral v)))
                        $ input  (pureIn  (CInt . fromIntegral))
                        $ result (pureRes fromIntegral)
                        ) compute
```

Construction, innermost first:

```
1) result (pureRes fromIntegral)
     :: Spec (IO CInt) (IO Int)

2) input (pureIn (CInt . fromIntegral)) $ <1>
     Pure Int → CInt. ThreadIn's base case fires (inner hi is `IO Int`).
     :: Spec (CInt -> IO CInt) (Int -> IO Int)

3) output (peekOut ...) $ <2>
     Allocates a `Ptr CInt`, runs the call, peeks the result.
     ThreadOut now sees an inner hi of `Int -> IO Int`, a function
     type. Its step instance fires, introducing a `\arg ->` lambda;
     the inner ThreadOut at `IO Int` then fires its base case, and
     the result type grows by one tuple slot.
     :: Spec (Ptr CInt -> CInt -> IO CInt) (Int -> IO (Int, Int))
```

Runtime, unfolded:

```haskell
toHighLevel <step3> compute
  = threadOut (withOutM (peekOut k))
              (\slot -> toHighLevel <step2> (spread slot compute))
  where k (CInt v) = pure (fromIntegral v)

  -- step instance of ThreadOut fires (one arg):
  = \arg -> threadOut (withOutM (peekOut k))
                      (\slot -> toHighLevel <step2> (spread slot compute) arg)

  -- base case fires:
  = \arg -> alloca $ \p -> do
              r  <- toHighLevel <step2> (compute p) arg -- runs input + result
              c  <- peek p                              -- read out-param
              hs <- k c                                 -- convert to Int
              pure (hs, r)
```

### Putting the type classes together

If the type classes type variables and rules were too confusing here's an
attempt to make it all make sense:

1. The spec is built from right to left.

Because `$` is right-associative, and he result type is the base case in
`ThreadIn` and `ThreadOut`. A chain like:

```haskell
input  withCStringIn
  $ output (peekOut k)
  $ result (pureRes fromIntegral)
```

is parenthesised as

```haskell
input withCStringIn (output (peekOut k) (result (pureRes fromIntegral)))
```

So the innermost combinator, `result` here, comes together first.
GHC builds the spec from there outward: `output` wraps the `result`, `input`
wraps the `output`.

2. The GADT rules recurse over `lo` and `hi`.

Each `In` and `Out` does two walks at once: `Spread` consumes from the head of
`lo`, and `ThreadIn`/`ThreadOut` recurses on `hi`. The walks are independent;
they just both fire for every `In` or `Out` you write.

3. `ThreadIn` introduces a Haskell argument; `ThreadOut` does not.

#### A combined example

Take a function that interleaves outputs with an input:

```c
int interleaved(int *out1, int n, int *out2);
/* writes n into *out1, n*2 into *out2, returns 0. */
```

```haskell
hsInterleaved :: Int -> IO (Int, (Int, Int))
hsInterleaved = toHighLevel ( output peekIntOut                     -- 1st C arg: out
                            $ input  (pureIn (CInt . fromIntegral)) -- 2nd C arg: in
                            $ output peekIntOut                     -- 3rd C arg: out
                            $ result (pureRes fromIntegral)         -- C return:  status
                            ) interleaved
  where peekIntOut = peekOut (\(CInt n) -> pure (fromIntegral n))
```

The construction walk (innermost first). At each step we show the
combinator's signature, the marshaller's type, the type-class
constraints that fire (with every type variable bound to a concrete
type), and the resulting `Spec`.

Step 1. `result (pureRes fromIntegral)`

```
  result               :: ResMarshaller c hs -> Spec (IO c) (IO hs)
  pureRes fromIntegral :: ResMarshaller CInt Int

  Type variables:
    c  = CInt
    hs = Int

  Result:
    <1> :: Spec (IO CInt) (IO Int)
```

Step 2. `output peekIntOut $ <1>`

```
  output      :: (Spread slot lo lo', ThreadOut slot hs rest rest')
              => OutMarshaller hs slot -> Spec lo' rest -> Spec lo rest'
  peekIntOut  :: OutMarshaller Int (Ptr CInt)

  From the marshaller:     hs   = Int
                           slot = Ptr CInt
  From the inner spec <1>: lo'  = IO CInt
                           rest = IO Int

  Spread (Ptr CInt) lo (IO CInt)
    fires the single-arg fallback instance Spread t (t -> lo) lo
    with t = Ptr CInt:
      lo = Ptr CInt -> IO CInt

  ThreadOut (Ptr CInt) Int (IO Int) rest'
    fires the base instance ThreadOut slot hs (IO r) (IO (hs, r))
    with r = Int:
      rest' = IO (Int, Int)

  Result:
    <2> :: Spec (Ptr CInt -> IO CInt) (IO (Int, Int))
```

**Step 3.** `input (pureIn (CInt . fromIntegral)) $ <2>`

```
  input         :: (Spread slot lo lo', ThreadIn slot rest)
                => InMarshaller hs slot -> Spec lo' rest -> Spec lo (hs -> rest)
  pureIn (CInt . fromIntegral) :: InMarshaller Int CInt

  From the marshaller:     hs   = Int
                           slot = CInt
  From the inner spec <2>: lo'  = Ptr CInt -> IO CInt
                           rest = IO (Int, Int)

  Spread CInt lo (Ptr CInt -> IO CInt)
    fires the single-arg fallback with t = CInt:
      lo = CInt -> Ptr CInt -> IO CInt

  ThreadIn CInt (IO (Int, Int))
    fires the base instance ThreadIn slot (IO r) with r = (Int, Int).
    No effect on the inner hi.

  Outer hi: hs -> rest = Int -> IO (Int, Int)

  Result:
    <3> :: Spec (CInt -> Ptr CInt -> IO CInt) (Int -> IO (Int, Int))
```

**Step 4.** `output peekIntOut $ <3>`

```
  output      :: (Spread slot lo lo', ThreadOut slot hs rest rest')
              => OutMarshaller hs slot -> Spec lo' rest -> Spec lo rest'
  peekIntOut  :: OutMarshaller Int (Ptr CInt)

  From the marshaller:     hs   = Int
                           slot = Ptr CInt
  From the inner spec <3>: lo'  = CInt -> Ptr CInt -> IO CInt
                           rest = Int -> IO (Int, Int)

  Spread (Ptr CInt) lo (CInt -> Ptr CInt -> IO CInt)
    fires the single-arg fallback with t = Ptr CInt:
      lo = Ptr CInt -> CInt -> Ptr CInt -> IO CInt

  ThreadOut (Ptr CInt) Int (Int -> IO (Int, Int)) rest'
    sees a function-shaped `rest`, so the step instance fires:
      ThreadOut slot hs (arg -> rest_inner) (arg -> rest_inner')
    with arg = Int and rest_inner = IO (Int, Int). It recurses:

      ThreadOut (Ptr CInt) Int (IO (Int, Int)) rest_inner'
        fires the base instance with r = (Int, Int):
          rest_inner' = IO (Int, (Int, Int))

    So:
      rest' = Int -> IO (Int, (Int, Int))

  Result:
    <4> :: Spec (Ptr CInt -> CInt -> Ptr CInt -> IO CInt)
                (Int -> IO (Int, (Int, Int)))
```

## Worked examples

This section exercises the design against a representative mix of C
patterns: a string-only call, a multi-slot matrix input, declarative
outputs, an effectful conversion, a callback, and the c2hs `+S`
buffer pattern. Each example shows the C signature, the generated
low-level binding (where helpful), and the user-facing wrapper.

### Closing the loop: the QR-code example

The introduction showed the boilerplate version of `basicDemo` with
`withCAString`, two `allocaArray`s, manual `unsafeFromPtr` and `fromBool`
around the call. With the DSL, the same call collapses to one spec:

```c
bool qrcodegen_encodeText(
    const char *text,
    uint8_t    *tempBuffer,
    uint8_t    *qrCode,
    enum qrcodegen_Ecc   errCorLvl,
    int         minVersion,
    int         maxVersion,
    enum qrcodegen_Mask  mask,
    bool        boostEcl);
```

Generated low-level:

```haskell
qrcodegen_encodeText :: PtrConst CChar
                     -> Ptr Word8
                     -> Ptr Word8
                     -> Qrcodegen_Ecc
                     -> CInt -> CInt
                     -> Qrcodegen_Mask
                     -> CBool
                     -> IO CBool
```

Wrapper:

```haskell
qrEncodeText :: String
             -> Qrcodegen_Ecc -> CInt -> CInt -> Qrcodegen_Mask -> Bool
             -> IO (Maybe (IncompleteArray Word8))
qrEncodeText text ecc vMin vMax qrMask boost = do
  (_scratch, (qrBuf, ok)) <- encode text ecc vMin vMax qrMask boost
  pure $ if ok then Just qrBuf else Nothing
  where
    bufLen = fromIntegral qrcodegen_BUFFER_LEN_MAX
    toBool = (/= 0)

    encode :: String
           -> Qrcodegen_Ecc -> CInt -> CInt -> Qrcodegen_Mask -> Bool
           -> IO ( IncompleteArray Word8         -- tempBuffer (scratch)
                 , (IncompleteArray Word8, Bool) -- (qrCode, status)
                 )
    encode = toHighLevel ( input  withCStringIn         -- const char *text
                         $ output (allocaBufOut bufLen) -- uint8_t *tempBuffer (scratch)
                         $ output (allocaBufOut bufLen) -- uint8_t *qrCode     (kept)
                         $ input  (pureIn id)           -- ecc level
                         $ input  (pureIn id)           -- minVersion
                         $ input  (pureIn id)           -- maxVersion
                         $ input  (pureIn id)           -- mask
                         $ input  (pureIn fromBool)     -- boostEcl
                         $ result (pureRes toBool)
                         ) qrcodegen_encodeText
```

Three things to notice:

1. The two `output`s are declarative: no manual `allocaArray`,
   no `peek`, no buffer-management boilerplate at the call site.
   The first `output` is the C-side scratch buffer, which we ignore
   on the Haskell side (the `_scratch` pattern).
2. The enums and version numbers are pure inputs: `pureIn id`
   passes them straight through, no marshalling needed.
3. The result is a `Maybe` because the spec returns a status
   `Bool` plus the QR buffer; we collapse them in the trailing line.

The wrapper is six declarative lines plus a one-line projection at
the bottom. Compared to three nested `with*` blocks and a hand-rolled
call in the original.

### A matrix input: `Slot3` in action

`Slot2` is the c2hs `&` pattern (one Hs value → 2 C args). `Slot3`
extends the same idea to three C arguments. A typical use is a matrix
that maps to `(rows, cols, data)`:

```c
double matrix_sum(int rows, int cols, const double *data);
```

```haskell
foreign import ccall unsafe "matrix_sum"
  c_matrix_sum :: CInt -> CInt -> PtrConst CDouble -> IO CDouble
```

A small Haskell-side type for the matrix:

```haskell
data Matrix a = Matrix
  { mRows :: Int
  , mCols :: Int
  , mData :: VS.Vector a -- row-major; length = rows * cols
  }
```

The marshaller fills three consecutive C slots from one `Matrix`:

```haskell
useAsMatrixIn :: VS.Storable a
              => InMarshaller (Matrix a) (Slot3 CInt CInt (PtrConst a))
useAsMatrixIn = InMarshaller $ \m k ->
  VS.unsafeWith (mData m) $ \p ->
    k (Slot3 (fromIntegral (mRows m))
             (fromIntegral (mCols m))
             (PtrConst.unsafeFromPtr p))
```

The wrapper reads in one line:

```haskell
matrixSum :: Matrix Double -> IO Double
matrixSum = toHighLevel ( input  useAsMatrixIn
                        $ result (pureRes realToFrac)
                        ) c_matrix_sum
```

`Slot3` is the same machinery as `Slot2`, just with one more
constructor argument. The `Spread (Slot3 a b c) (a -> b -> c -> lo) lo`
instance does the three-way uncurrying inside `toHighLevel`.

### An effectful result: wrapping `malloc`

`malloc`, which returns a raw pointer, needs to be copied into a `ByteArray`.
`ResMarshaller` is effectful by construction, and `ioRes` lifts an arbitrary
`c -> IO hs`:

```c
void *ll_malloc(int n);
```

```haskell
foreign import ccall unsafe "ll_malloc"
  c_ll_malloc :: CInt -> IO (Ptr Word8)

bufferToByteArray :: Int -> Ptr Word8 -> IO ByteArray
bufferToByteArray n p = do
  arr <- newByteArray n               -- copy the pointer's bytes,
  copyPtrToMutableByteArray arr 0 p n -- then free the C-side pointer
  free p
  unsafeFreezeByteArray arr

hsMalloc :: Int -> IO ByteArray
hsMalloc n = toHighLevel ( input  (pureIn fromIntegral)
                         $ result (ioRes (bufferToByteArray n))
                         ) c_ll_malloc n
```

### A callback: `FunPtr` inputs

A C API that takes a callback gives the design a stress test:
`FunPtr`s have a *creation* side (a wrapper that lifts a Haskell
function into a `FunPtr`) and a *teardown* side
(`freeHaskellFunPtr`). The runtime already ships the bracketed
combination as
[`HsBindgen.Runtime.Prelude.withFunPtr`][runtime-withfunptr],
keyed on a `ToFunPtr` instance that `hs-bindgen` generates for
common callback signatures:

```haskell
withFunPtr :: ToFunPtr a => a -> (FunPtr a -> IO b) -> IO b
withFunPtr x = bracket (toFunPtr x) freeHaskellFunPtr
```

That's exactly the shape an `InMarshaller` wants, so the marshaller is one
line:

```haskell
funPtrIn :: ToFunPtr a => InMarshaller a (FunPtr a)
funPtrIn = InMarshaller withFunPtr
```

The wrapper, for a typical "register a callback" C API:

```c
typedef void (*callback_t)(int);
void register_callback(callback_t cb);
```

```haskell
type Callback = CInt -> IO ()
-- A `ToFunPtr Callback` instance is generated by hs-bindgen.

foreign import ccall unsafe "register_callback"
  c_register_callback :: FunPtr Callback -> IO ()

hsRegisterCallback :: Callback -> IO ()
hsRegisterCallback = toHighLevel ( input funPtrIn
                                 $ discardRes
                                 ) c_register_callback
```

A one-line marshaller built on top of the runtime's `withFunPtr`
gives the user the convention "supply a Haskell function; the DSL
handles the `FunPtr` creation and cleanup".

[runtime-withfunptr]: hs-bindgen-runtime/src/HsBindgen/Runtime/Internal/FunPtr/Class.hs

## Reading error messages

The DSL's surface is small, but it's still built on top of a
type-class infrastructure (`Spread`, `ThreadIn`, `ThreadOut`). When
the user gets something wrong, the type-checker has to point at it,
and we should know what those messages look like. This section
catalogues the common failure modes by *actually compiling broken
specs* and showing GHC's verbatim output.

The full set of broken specs is reproducible — drop the
[scratch file at the end of this section](#scratch-file-for-reproducing-the-errors)
into any module that imports `HsBindgen.Runtime.HighLevel.Spec` and
compile with `cabal exec -- ghc -fno-code`.

### Failure 1: forgot an `input`

```haskell
err1 :: String -> IO Int
err1 = toHighLevel (result (pureRes fromIntegral)) c_strlen
```

GHC says:

```
error: [GHC-83865]
    • Couldn't match type: IO hs1
                     with: String -> IO Int
      Expected: Spec (IO c0) (String -> IO Int)
        Actual: Spec (IO c0) (IO hs1)
    • In the first argument of ‘toHighLevel’, namely
        ‘(result (pureRes fromIntegral))’

error: [GHC-83865]
    • Couldn't match expected type: IO c0
                  with actual type: PtrConst CChar -> IO CSize
    • Probable cause: ‘c_strlen’ is applied to too few arguments
```

**How to read it.** Two complementary errors. The first says the
spec produces `IO hs1` but the user signature expects `String -> IO
Int`. The second says the C function expected `IO c0` (i.e. fully
applied) but is still a function. Diagnosis: the spec doesn't
consume the `PtrConst CChar` input — add `input withCStringIn $ `
in front of the `result`.

### Failure 2: wrong slot type at the head

```haskell
err2 :: Int -> IO Int
err2 = toHighLevel ( input  (pureIn fromIntegral :: InMarshaller Int CInt)
                   $ result (pureRes fromIntegral)
                   ) c_strlen
```

GHC says:

```
error: [GHC-39999]
    • No instance for ‘Spread CInt (PtrConst CChar -> IO CSize) (IO c0)’
        arising from a use of ‘input’
        (maybe you haven't applied a function to enough arguments?)
```

**How to read it.** GHC names `Spread` and shows both types it
tried to unify: the marshaller's slot is `CInt`, but the C function
starts with `PtrConst CChar`. Diagnosis: wrong marshaller for this
position (or wrong position for this marshaller).

### Failure 3: multi-slot marshaller against a single-arg function

```haskell
err3 :: ByteString -> IO Int
err3 = toHighLevel ( input  useAsByteStringLenIn
                   $ result (pureRes fromIntegral)
                   ) c_strlen
```

GHC says:

```
error: [GHC-39999]
    • No instance for ‘Spread (Slot2 (PtrConst CChar) CSize)
                              (PtrConst CChar -> IO CSize)
                              (IO c0)’
        arising from a use of ‘input’
```

**How to read it.** `Spread` is the bridge between a slot value and
the curried C args it should fill. Here the slot is `Slot2 (PtrConst
CChar) CSize` (asks for two args), but the C function only has
`PtrConst CChar` at the head and reaches `IO CSize` after one arg.
Diagnosis: the marshaller wants more C args than the function has at
this position.

### Failure 4: too many inputs for the C signature

```haskell
err4 :: String -> String -> String -> String -> IO Int
err4 = toHighLevel ( input  withCStringIn
                   $ input  withCStringIn
                   $ input  withCStringIn
                   $ input  withCStringIn
                   $ result (pureRes fromIntegral)
                   ) c_strncmp
```

GHC says:

```
error: [GHC-39999]
    • No instance for ‘Spread (PtrConst CChar) (CSize -> IO CInt) lo'0’
        arising from a use of ‘input’
```

**How to read it.** After three inputs have consumed
`PtrConst CChar`, `PtrConst CChar`, and `CSize`, the inner spec sees
the C function's remaining type as `IO CInt`. The fourth `input`
then asks GHC to find an instance `Spread (PtrConst CChar) (CSize ->
IO CInt) lo'0` — to consume yet another `PtrConst CChar` — but the
remaining low-level type starts with `CSize`, not `PtrConst CChar`.
Diagnosis: one input too many.

### Failure 5: result type mismatch

```haskell
err5 :: String -> IO Int
err5 = toHighLevel ( input  withCStringIn
                   $ result (pureRes ((!! 0) . show))
                   ) c_strlen
```

GHC says:

```
error: [GHC-83865]
    • Couldn't match type ‘Char’ with ‘Int’
      Expected: CSize -> [Int]
        Actual: CSize -> String
```

**How to read it.** This is just a plain Haskell type mismatch
inside the user-supplied result function (`(!! 0) . show` returns
`Char`, but the wrapper claims `IO Int`). The DSL's type-class
machinery doesn't appear at all. Diagnosis: fix the result
function's type.

### Failure 6: passed a marshaller where a spec was expected

```haskell
err6 :: String -> IO Int
err6 = toHighLevel ( input withCStringIn
                   $ pureRes fromIntegral
                   ) c_strlen
```

GHC says:

```
error: [GHC-83865]
    • Couldn't match expected type: Spec (IO CSize) (IO Int)
                  with actual type: ResMarshaller c4 hs0
```

**How to read it.** `input` expects an inner `Spec`, not a
`ResMarshaller`. GHC names both types. Diagnosis: forgot the
outer `result` around the `pureRes`.

### Verdict

The error messages are tractable. The names that show up in the
errors — `Spread`, `Slot2`, `Spec`, `InMarshaller`, `ResMarshaller`
— are the same names the user wrote in their spec, so the messages
point back to the user's code rather than disappearing into
elaborated internals. The one mild irritation is the secondary
"Ambiguous type variable" errors that follow many of the primary
errors (a result of `fromIntegral`'s polymorphism plus an unresolved
inner constraint); those will go away once the primary error is fixed.

`TypeError`-clad instances would convert "no instance for `Spread
(MultipleArgs N) ...`" into a hand-written sentence like "this
marshaller wants N C arguments at this position; the function only
has M". We have not added those yet, and the current messages are
clear enough that we recommend deferring the work; see § Open
questions, risks, caveats.

## Automatic generation of high-level bindings

The DSL above is fully explicit: the user writes every marshaller for
every position. That's the right default — it's the layer where the
user has full control. But many bindings have a "boring" shape where
every position would just use the obvious default marshaller; for
those, an opt-in inference shim is convenient.

We propose extending the design with two layers on top of the GADT —
**defaults** and **full inference** — neither of which removes any
expressiveness from the explicit layer.

### Layer 1: defaults

Two small type classes name the "default marshaller for type X" idea:

```haskell
class HasInDefault hs where
  type InSlot hs :: Type
  defaultIn :: InMarshaller hs (InSlot hs)

class HasResDefault c hs where
  defaultRes :: ResMarshaller c hs
```

Instance examples:

```haskell
instance HasInDefault String where
  type InSlot String = PtrConst CChar
  defaultIn = withCStringIn

instance HasInDefault Bool where
  type InSlot Bool = CBool
  defaultIn = pureIn fromBool

instance HasInDefault ByteString where
  type InSlot ByteString = Slot2 (PtrConst CChar) CSize
  defaultIn = useAsByteStringLenIn

instance Integral c => HasResDefault c Int where
  defaultRes = pureRes fromIntegral
```

A pair of value-level aliases that the user writes at the call site:

```haskell
autoIn  :: HasInDefault  hs  => InMarshaller  hs (InSlot hs)
autoIn  = defaultIn

autoRes :: HasResDefault c hs => ResMarshaller c hs
autoRes = defaultRes
```

With these in scope, a wrapper that uses defaults but keeps the spec
visible:

```haskell
hsStrlen :: String -> IO Int
hsStrlen = toHighLevel ( input  autoIn
                       $ result autoRes
                       ) c_strlen
```

Override one position by replacing `autoIn` with the explicit
marshaller — no boilerplate:

```haskell
hsStrlenUsingLen :: String -> IO Int
hsStrlenUsingLen = toHighLevel ( input  useAsByteStringLenIn  -- override
                               $ result autoRes
                               ) c_strncmp_first_n_bytes_or_similar
```

### Layer 2: full inference

A walker class consumes the high-level signature `hi` and the
low-level signature `lo` together, picking defaults at each position:

```haskell
class AutoSpec hi lo where
  autoSpec :: Spec lo hi

-- Base case: both signatures reach IO.
instance HasResDefault c hs
      => AutoSpec (IO hs) (IO c) where
  autoSpec = result defaultRes

-- Recursive case: an input argument.
instance ( HasInDefault hs
         , Spread (InSlot hs) lo lo'
         , ThreadIn (InSlot hs) rest
         , AutoSpec rest lo'
         )
      => AutoSpec (hs -> rest) lo where
  autoSpec = input defaultIn autoSpec
```

The Tier-3 shim is one line:

```haskell
hl :: AutoSpec hi lo => lo -> hi
hl lo = toHighLevel autoSpec lo
```

Side-by-side, the three tiers look like this:

```haskell
-- Tier 1: fully explicit, per-call customisation
hsStrlen :: String -> IO Int
hsStrlen = toHighLevel ( input  withCStringIn
                       $ result (pureRes fromIntegral)
                       ) c_strlen

-- Tier 2: defaults where they fit, explicit where they don't
hsStrlen' :: String -> IO Int
hsStrlen' = toHighLevel ( input  autoIn
                        $ result autoRes
                        ) c_strlen

-- Tier 3: full inference (when the defaults suffice)
hsStrlen'' :: String -> IO Int
hsStrlen'' = hl c_strlen
```

### Honest trade-offs

The defaults layer brings back the error-message risk discussed in
§ Reading error messages. The Tier-3 `hl` is the worst case: a
missing `HasInDefault` instance produces a wall of failed
constraints, with no `Spread`/`Slot` names to anchor the diagnosis
because elaboration never got that far.

We recommend shipping the explicit DSL as the default and treating
defaults / Tier-3 inference as opt-in. Three reasons:

1. The explicit form is more discoverable for new users — the
   wrapper's body shows exactly what's happening.
2. The explicit form has good error messages (§ Reading error
   messages).
3. The defaults are only a win when the user has many bindings with
   the same shape; for one-off wrappers, picking a marshaller is no
   harder than picking a default and then overriding.

The auto-generation layer is therefore framed as an addition to the
explicit DSL, not a replacement.

## Open questions, risks, caveats

Where the design might still need work, in priority order.

### `TypeError` polish for the type classes

The error messages in §[Reading error messages](#reading-error-messages)
are tractable but not pretty. A `TypeError`-clad instance head per
common failure mode (missing `Spread` for a particular slot/arity,
`Done` on a non-`IO` low-level type) would convert raw class-resolution
failures into hand-written sentences. We have not done this yet; the
current messages are good enough that the polish is deferable,
especially since the explicit-DSL surface keeps the user close to
where the error fires.

### Flat result tuples for ≥3 outputs

Each `output` prepends one component to the inner `IO`'s payload, so
three outputs and a result yield `IO (a, (b, (c, d)))`. For two
outputs this is unobtrusive; from three onward the nested-pair
shape is mildly awkward at the call site.

The proposed remedy is a `Results<N>` data family symmetric with
`Slot<N>`, accompanied by a final `flatten` pass on the inner
`IO`'s payload at `Done` time. We have not designed this yet —
keeping it open as a focused polish item once a real binding hits
the awkwardness.

### Generator-side support

The compiler currently emits the low-level binding only. Milestone-3
work is to have it also emit a `Spec` value (using the defaults from
§[Automatic generation of high-level bindings](#automatic-generation-of-high-level-bindings))
which the user can either use directly or override per position. The
GADT proposed here is the natural target; no design change is
expected on the runtime side.

### Performance baseline

The bracket-style `withIn :: hs -> (slot -> IO r) -> IO r` composes
cleanly but adds a closure allocation per argument unless GHC
inlines aggressively. We expect to ship `INLINE` pragmas on
`spread` / `threadIn` / `threadOut` / `toHighLevel` for the same
reason `Call.hs` already does — and to verify via Core dump that the
generated wrapper is identical to a hand-written one. The baseline
is the hand-written code; we should not be slower than that.

### `Spread`'s module home

`Spread` currently lives in `HsBindgen.Runtime.HighLevel.Call` (the
round-1 module). The `Slot<N>` instances live in
`HsBindgen.Runtime.HighLevel.Spec`. The class and its data instances
in different modules is non-idiomatic — but moving `Spread` to
`Spec.hs` would create an import cycle (round-1's `HighLevel` class
uses `Spread`). The cleanest fix is to move `Spread` into a small
shared module that both `Call.hs` and `Spec.hs` import. Deferred
until we decide the round-1 / round-2 module layout.

## Summary

The proposal is a small DSL built around three types of marshaller
values (input, output, return), a three-constructor `Spec lo hi`
GADT, and one interpreter `toHighLevel`. The user reads a spec
top-down in C-argument order; the compiler builds it bottom-up
because `$` is right-associative; and two small threading classes
defer each marshaller's bracket past whatever Haskell arguments the
inner spec has accumulated. The DSL covers the c2hs hook patterns
(per-call marshallers, the `&` multi-slot pattern, declarative
outputs), supports effectful conversion in the result, supports
nullable pointers via `Maybe`, and supports callbacks via `FunPtr`
marshallers. An opt-in defaults layer and a Tier-3 `hl` shim sit on
top for the common shapes. The full machinery — three classes, three
records, one GADT, one interpreter — fits in one module.
