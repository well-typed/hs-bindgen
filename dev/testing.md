# Testing Guide

## Running Tests

### Fast Mode

Use `--fast` (`-f`) to skip slow tests (fixture compilation and unsafe golden
tests) for quicker development iteration:

```bash
cabal run test-hs-bindgen -- --fast
```

### Haddock Mode

Use `--haddock` to generate browsable Haddock documentation for all fixtures:

```bash
cabal run test-hs-bindgen -- --haddock
```

Output goes to `dist-newstyle/fixtures/haddock-fixtures/`.

### Debug Mode

Use `--debug` (`-v`) to print all trace messages during golden test execution:

```bash
cabal run test-hs-bindgen -- --debug
```

### Golden Test Updates

When tests fail due to expected output changes:

```bash
cabal run -- test-hs-bindgen --accept
```

## Fixture Compilation

The PP and TH fixture compilation tests verify that generated Haskell code
compiles (and optionally generates Haddock documentation).

### How it works

Fixtures are compiled directly via `ghc` (or `haddock --ghc` in haddock mode).
The GHC environment file generated by `cabal build` (via
`write-ghc-environment-files: always` in `cabal.project`) provides package
visibility.

Each fixture is compiled as a separate `ghc --make` process. Invocations run
in parallel (bounded by available cores), and each returns its own exit code
and stderr for per-fixture error attribution.

### Why not `cabal build`?

With ~340 internal libraries per package, `cabal build -j` triggers
non-deterministic `ghc-pkg` failures: concurrent registrations race on the
shared package database
([#1813](https://github.com/well-typed/hs-bindgen/issues/1813),
[GHC #13194](https://gitlab.haskell.org/ghc/ghc/-/issues/13194)).

### Troubleshooting: stale GHC environment file

The `.ghc.environment.*` file references package databases under the shared
cabal store (`~/.cabal/store` or `$CABAL_DIR/store`).  If the store is cleaned
(or a different GHC is now on `PATH`) after the file was written, the paths no
longer exist and fixture compilation fails.

The test suite detects this and fails with a clear message listing the missing
paths.  Regenerate the env file by running `cabal build all`.

## Continuous Integration

### Debugging CI Failures

Add a tmate session to the CI workflow:

```yaml
- name: Setup tmate session
  uses: mxschmitt/action-tmate@v3
```

This provides an SSH address to debug the CI runner interactively.
